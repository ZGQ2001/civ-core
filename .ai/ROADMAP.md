# 产品路线图

> 只看产品要做什么、为什么按这个顺序做、做完后用户怎么用。不写代码，不写架构，不写当前进度。

---

## 一、要解决的问题

土木检测公司的内业工作流：

```
原始数据（Excel）→ 查规范判定 → 画曲线 → 填报告 → 转 PDF
```

每一步都是人手做。一个锚杆项目几十根杆，同一组公式反复算、同一套表格反复填。规范条文是固定的、报告格式是固定的——这种机械重复不应该由人来做。人应该做的是审核结果和做判定。

**这个软件要解决的核心痛点**：把机械重复的步骤自动化，让人只做决策。

---

## 二、工程特点与行业风险

**跟普通软件不同的地方**：

**1. 出错代价是工程事故。**
一份锚杆报告判错了合格/不合格，现场可能按错的参数施工。所以每一步必须可追溯：哪组数据、走了哪个规范公式、得出什么判定结果——能往回查。程序不能是黑盒。

**2. 报告格式由甲方定。**
每个甲方有自己的 Word 模板。行高列宽、合并单元格、签字栏位置全不一样。程序不能假设"表头在第三行"——它得能适应任何甲方给的模板。

**3. 规范是活的。**
GB 50086 和 JGJ 106 的判定公式不同。同一组数据，换规范就可能从合格变不合格。计算和规范必须严格绑定，选什么规范出什么结果，不能混。

**需要规避的风险**：

| 风险 | 怎么做 |
|------|--------|
| 规范用错 | 锁定规范版本，报告里写明判定依据 |
| A 项目数据串到 B 项目 | 工作区隔离，一个项目一个文件夹 |
| 公式实现错误 | 所有判定路径全覆盖测试 |
| 用户看不懂程序在干嘛 | 每一步有日志反馈，判定结果附公式和中间值 |

## 三、与其他工程软件的区别

**传统土木软件（理正、PKPM）**：人操作软件，一个按钮一个功能。人点「计算」→ 人等结果 → 人点「出报告」→ 人等报告。

**civ-core**：两种模式共存。

- **手动模式**：人操作软件，跟传统软件一样，走完完整流程。
- **Agent 模式**：人变成决策者。人说「这批锚杆用 GB 50086 出报告」，Agent 自己找数据、选模版、填报告、转 PDF。人只需要在关键判定点确认——「3 号锚杆不合格，继续吗？」——点继续就行。

同一套工具，既可以用手点，也可以用嘴说。

---

## 四、第一阶段：制作各个模块

**目标**：每个检测类型有一个能独立跑通的工具页。

**做什么**：
- 里氏硬度工具页
- 锚杆抗拔工具页
- 曲线绘制工具页
- PDF 合并/拆分工具页
- Word 转 PDF 工具页
- 钻芯法工具页
- 回弹法工具页
- 内置模版制作器

**每个工具页独立可用**。一个内业员即便不走完整流程，也能单独用某个工具完成一项工作：只画曲线、只转 PDF、只算锚杆。每个工具页都做到「打开 → 填参数 → 点运行 → 出结果」，不依赖其他工具页。

**为什么先做模块**：零件没造好，别拼流水线。每个模块稳定了，后面串流程才有可靠的基础。

**验收条件**：每个工具页能独立完成自己的那一步，结果正确且可追溯。

---

## 五、第二阶段：人工按流程出完整报告

**目标**：把第一阶段做好的模块串成一条线。用户手动走完「原始数据 → 完整报告」的全流程。

**流程**：

```
新建项目 → 上传数据 → 选检测类型跑计算 → 选模版跑曲线填报告 → 导出 PDF
```

**每步都是人点的**。程序不自动跳下一步——因为每步跑完，人要看结果对不对。

**模版制作器在这里发挥关键作用**：用户在第一阶段做好的模版，到第二阶段直接加载。换甲方 = 换模版，不改代码。

**第一阶段和第二阶段的关系**：第一阶段造零件，第二阶段把零件拼成流水线。但工具页仍然独立可用——你仍然可以只打开曲线工具画张图就走。多出来的「项目模式」入口才是走完整流程的。

**验收条件**：一个内业员打开软件，从新建项目到导出完整报告和 PDF，只靠人点按钮走完，中间不碰 Excel 和 Word。

---

## 六、第三阶段：引入 AI Agent

**目标**：在手动模式之上加一层 Agent 模式。人变成决策者，Agent 替代手工操作。

**两种模式共存，不互斥**：

```
手动模式：人点按钮 → 程序执行 → 人看结果 → 人点下一个按钮

Agent 模式：人说一句话 → Agent 拆步骤调用程序 →
           中间结果推给人 → 人做判定 → Agent 继续
```

**Agent 模式的使用场景**：

- 人：「这批数据是锚杆抗拔，用 GB 50086，模版用上次那个」
- Agent：自动读 Excel → 跑计算 → 生成曲线 → 填报告
- Agent：推送中间结果——「3 号锚杆弹性位移量超限不合格，其余 47 根合格。继续？」
- 人：「继续」
- Agent：出完整报告 → 转 PDF
- 人只做了两次决策：接受判定、确认继续

**手动模式和 Agent 模式谁优先**：手动是底线。Agent 模式能跑的前提是手动模式已经稳定——流程跑稳了，Agent 只是把「人点按钮」换成了「Agent 调接口」。如果 Agent 出错，随时切回手动继续。

**为什么这是第三阶段**：Agent 需要每条工具线和完整流程都跑稳。不能边修 bug 边让 Agent 调——Agent 不会判断是 bug 还是正常结果，它会照单全收。第二阶段的完整流程跑通了，才敢把方向盘交给 Agent。

**验收条件**：人说一句话出完整报告，全程人只在判定点做决策，不碰任何按钮。

---

## 七、第四阶段：打包发布 + 持续填充

**目标**：稳定版本打成安装包，持续加新检测类型。

**打包**：一个 exe，双击装好就能用。不依赖 Python 环境、不依赖 .NET SDK。

**持续填充**：
- 更多检测类型（钻芯、回弹、超声、静载……）
- 更多规范版本（JGJ 106、CECS 03……）
- 更多甲方模板适配
- 规范更新（GB 修订后更新数据库）

**这个阶段没有终点。** 只要还有没覆盖的检测类型，就可以一直加。

**验收条件**：新用户安装 5 分钟出第一份报告。老用户一句话出报告。
