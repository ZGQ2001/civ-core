# 审查报告：土木工程师无法理解的报错消息

**日期**：2026-05-24  
**审查项**：全量代码中的用户可读性——报错消息、UI 文案是否能让土木工程师理解  
**审查方法**：逐层扫描 RPC handler → sidecar 错误传播 → 前端错误展示

---

## 发现

### 🔴 严重：C# Handler 报错全是程序员术语（14 处）

所有 C# handler 抛出的 `ArgumentException` 经 JsonRpcServer 包装后，用户看到的是：

> `ArgumentException: params 必须是 object`

土木工程师不知道 `object`、`array`、`params_by_batch` 是什么。这些是程序内部术语，不应暴露给用户。

| `LeebHandlers.cs` | 41 | `"缺 input_xlsx"` | 暴露内部参数名 |
| `LeebHandlers.cs` | 99 | `"params 必须是 object"` | 同上 |
| `LeebHandlers.cs` | 103 | `"缺 path"` | 暴露内部参数名 |
| `AnchorHandlers.cs` | 37,60,80 | `"params 必须是 object"` | 3 处同问题 |
| `AnchorHandlers.cs` | 41,64,84 | `"缺 input_xlsx"` / `"缺 output_xlsx"` | 暴露内部参数名 |
| `AnchorHandlers.cs` | 156 | `"params_by_batch 必须是 object（key=batch_id）"` | 土木完全不懂 |

### 🔴 严重：JsonRpcServer 异常类型名泄漏

`JsonRpcServer.cs:87` 和 `server.py:99` 的错误处理把异常类型名拼接进消息：

```
code: -32603, message: "ArgumentException: params 必须是 object"
code: -32603, message: "ValueError: 名称不能为空"
```

土木工程师看到 `ArgumentException`、`ValueError` 这类程序异常类型名，完全无法理解。

### 🔴 严重：前端无错误消息翻译层

- `rpc.ts:16` — RPC 调用无错误格式化，`invoke('rpc_call')` 抛出的异常直接冒泡
- `App.tsx:108` — `后端连接失败: ${String(e)}` — 错误原样透传
- `FileTree.tsx:526,550,564,598,605` — 所有 `alert()` 全部 `String(e)` 原样弹窗
- `plot_curves/Page.tsx:530,547,554,569,580,591` — 同上

### 🟡 中等：Python handler 报错相对好，但有零散问题

| 文件 | 行号 | 当前消息 | 问题 |
|------|------|----------|------|
| `server.py` | 75 | `"JSON 解析失败"` | 土木不知道 JSON 是什么 |
| `server.py` | 78 | `"请求必须是 JSON 对象"` | 同上 |
| `server.py` | 91 | `"未知 method: {method}"` | "method" 是程序概念 |
| `plot_curves.py` | 168 | `{"ok": False, "message": "..."}` | message 内容是好的，但传播链短 |
| `word2pdf.py` | 52,58,70 | `"文件不存在"/"读文件大小失败"/"解析失败"` | 这些还可以，message 本身是中文 |

### 🟢 良好：files.py 的错误消息

`files.py` 的报错是标杆——土木工程师能直接看懂：
- `"名称不能为空"`
- `"名称首尾不能含空格"`
- `"父目录不存在：xxx"`
- `"超过 5 分钟的删除不支持在 App 内撤销，请前往系统回收站手动还原。"`

---

## 根因

1. **C# handler 抛出的异常直接作为用户可见消息**——JsonRpcServer 只做 `$"{e.GetType().Name}: {e.Message}"` 拼接，没有「内部错误码 → 用户友好消息」的翻译层
2. **前端没有统一错误处理**——每个组件各自 `catch(e) → alert(String(e))`，没有机会做术语翻译
3. **没有区分「程序员看的错误」和「工程师看的错误」**——当前所有错误信息一刀切

## 评级

| 层 | 严重度 | 说明 |
|----|--------|------|
| C# handlers | 🔴 | 14 处报错含程序员术语 |
| JsonRpcServer | 🔴 | 异常类型名泄漏 |
| 前端 rpc.ts | 🔴 | 无翻译层 |
| Python handlers | 🟡 | 大部分可读，少数有问题 |
| Python files.py | 🟢 | 标杆级 |

---

## 建议修复策略

1. **C# handler 层**：所有 `throw new ArgumentException(...)` 改用中文用户语言，不要提及 object/array/参数名
2. **JsonRpcServer**：去掉 `e.GetType().Name` 前缀，只传 message
3. **前端**：在 rpc.ts 或统一的 error boundary 加一层错误消息翻译（后端错误码 → 用户提示）
