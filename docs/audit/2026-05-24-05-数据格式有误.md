# 审查报告：数据格式有误 & 损坏文件处理

**日期**：2026-05-24  
**审查项**：Excel 列缺失、数据类型错误、空单元格、损坏文件、格式不匹配——是否被拦截并给出可理解的报错  
**审查方法**：逐 reader 审查单元格级校验、异常传播链路

---

## 发现

### 🟢 LeebExcelReader：逐单元格校验（8 项检查）

| 检查点 | 行号 | 行为 |
|--------|------|------|
| 非数字序号行（子表头） | 105 | 跳过 ✅ |
| C 列非数字行 | 112 | 跳过 ✅ |
| HL 值为空 | 137 | 抛异常，指明行号+列号 ✅ |
| HL 值非数字 | 141 | 抛异常，指明行号+列号+实际值 ✅ |
| 构件名为空 | 153 | 抛异常 ✅ |
| 厚度值为空 | 157 | 抛异常 ✅ |
| 厚度值 ≤0 或非数字 | 159 | 抛异常 ✅ |

**评级：生产级。** 连「子表头」「空行」等 Excel 常见脏数据都做了容错。

### 🟢 AnchorExcelReader：列头匹配+逐行校验

| 检查点 | 行号 | 行为 |
|--------|------|------|
| 文件不存在 | 18 | FileNotFoundException ✅ |
| 缺失 batch_id 列 | 27 | RequireColumn 抛异常 ✅ |
| 缺失锚杆编号列 | 28 | RequireColumn 抛异常 ✅ |
| 位移读数列缺失 | 30-31 | RequireColumn 抛异常 ✅ |
| 批次列/编号列为空 | 41 | 跳过空行 ✅ |
| 批次/锚杆 ID 为空 | 45-48 | 抛异常，指明行号 ✅ |

### 🟡 所有异常最终都变成 -32603 Internal error

```
User sees: {"code": -32603, "message": "ArgumentException: 行 5 厚度无效：abc"}
```

RPC 层的错误包装（`JsonRpcServer:87` + `server.py:99`）统一把 handler 异常包成 Internal error。对于**数据格式错误**，更好的做法是用 RPC 错误码 `-32602 Invalid params` 并给用户清晰的提示。

### 🟡 损坏文件处理——依赖 ClosedXML 底层异常

文件不是合法 xlsx（如后缀名改了但内容是 PDF）时，`new XLWorkbook(path)` 抛出的异常信息是 ClosedXML 内部的，用户看到的是类似：

> `FileFormatException: The file is not a valid package`

而不是「该文件不是有效的 Excel 文件，请检查文件是否损坏」。

### 🟢 JSON-RPC 协议层有解析兜底

`JsonRpcServer:48-55`：非法 JSON → `-32700 Parse error`。前端收到的是 JSON-RPC 标准错误码，不会崩溃。

## 评级

| 问题 | 严重度 | 说明 |
|------|--------|------|
| 单元格级数据校验 | 🟢 | Leeb + Anchor 都很完善 |
| 错误最终表现 | 🟡 | 所有数据错误都变成 -32603，应该用 -32602 |
| 损坏文件报错 | 🟡 | ClosedXML 原始异常对用户不友好 |
| RPC 协议层解析 | 🟢 | JSON 解析错误有标准兜底 |

---

## 建议

1. **区分 Internal error 和 Invalid params**：handler 抛出的 `ArgumentException` 映射为 `-32602 Invalid params`，真正的内部异常（如 ClosedXML 崩溃）才用 `-32603`
2. **文件格式校验提前**：在 `new XLWorkbook()` 之前，用 `File.ReadAllBytes` 检查文件头（xlsx 是 ZIP，头 4 字节 `PK\x03\x04`）
3. **错误消息翻译**：数据格式错误的 message 已有具体信息（行号、列号、值），配合审查 #01 的术语翻译层可以让用户看懂
