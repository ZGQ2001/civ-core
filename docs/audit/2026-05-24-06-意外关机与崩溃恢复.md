# 审查报告：意外关机 & Sidecar 崩溃恢复

**日期**：2026-05-24  
**审查项**：写文件时断电是否损坏数据、sidecar 进程崩溃后能否自愈、有无自动保存  
**审查方法**：逐文件检查写入模式 (SaveAs vs 原子替换)、Rust sidecar 管道的超时+恢复机制

---

## 发现

### 🔴 C# 侧全部直接覆盖写入——Power loss 必损坏（4 处）

所有 C# handler 的输出都走 `wb.SaveAs(path)` 直接覆盖：

| 位置 | 方法 | 写入方式 |
|------|------|----------|
| `XlsxHandlers.cs:93` | 里氏报告表 | `SaveAs(outputPath)` — 直接覆盖 |
| `AnchorHandlers.cs:141` | 锚杆分析+报告 | `SaveAs(outPath)` — 直接覆盖 |
| `AnchorTemplateWriter.cs:78` | 锚杆模板 | `SaveAs(path)` — 直接覆盖 |
| `AnchorHandlers.cs:127` | 已存在文件追加 | `new XLWorkbook(outPath)` → `SaveAs` — 读-改-写无保护 |

**风险**：正在写 xlsx 时断电——文件变成半截 ZIP 包，无法打开。如果文件之前已经存在（XlsxHandlers 是追加 sheet 场景），老内容也丢了。

**对比**：Python 侧的 `plot_curves`、`pdf_tools`、`preset_manager` 全部走 `atomic_writer`——写临时文件 → 成功后 `os.replace` 原子替换。断电也只丢临时文件，原文件完好。

### 🟢 Rust sidecar 管道可靠性：已修复（原 05-22 审计 R1+R2）

上次审计的 3 个 Rust 🔴 中有 2 个已修：

| 原问题 | 05-22 状态 | 当前状态 |
|--------|-----------|----------|
| `read_line` 无超时 | 🔴 未修 | ✅ 30s 超时 (sidecar.rs:138) |
| stderr 从不读取 | 🔴 未修 | ✅ drain task 按行 log (sidecar.rs:74-89) |
| sidecar 崩溃不重启 | 🔴 未修 | 🟡 检测+fast-fail，但无重启 |

### 🟡 Sidecar 崩溃检测到位，但需手动重启应用（sidecar.rs:107-112）

```rust
if !self.alive.load(Ordering::Acquire) {
    return Err(anyhow!("{} sidecar 已死（进程崩溃或超时），请重启应用", self.name));
}
```

- `alive` 标志在超时/EOF/stdout 读失败时置 false
- 后续调用立即 fast-fail，不排队等死锁 ✅
- 但提示用户「请重启应用」——没有自动重启 sidecar 进程

**影响**：如果 sidecar 崩溃，当前工具页的操作会报错，用户看到「请重启应用」提示，必须手动关掉 Tauri 再开。不算严重（桌面应用重启很快），但不够优雅。

### 🟢 无自动保存——但当前无实时编辑场景

civ-core 是「输入 Excel → 一键计算 → 写结果文件」的批处理模式，没有「用户编辑中需要自动保存」的长时编辑场景。不需要 auto-save。

### 🟢 网络中断无影响

civ-core 是纯本地桌面应用。frontend ↔ Tauri ↔ sidecar 全走本地进程间通信（stdin/stdout pipes）。唯一网络依赖是可能加载在线规范数据库（如果有），但目前 `standards.db` 是本地 SQLite。

## 建议

1. **C# 侧加原子写入**：参考 Python `atomic_writer` 模式，用 `Path.GetTempFileName()` + `File.Move` (同卷原子) 实现
2. **Sidecar 自动重启**：在 `call()` 检测到 alive=false 时，尝试 `spawn` 新 sidecar 替换 Arc 内的实例
3. **前端错误提示优化**：sidecar 崩溃时前端展示的「请重启应用」可以改成带一键重连按钮
