# 审查报告：大文件 & 内存不足场景

**日期**：2026-05-24  
**审查项**：大量数据/图片同时处理是否 OOM、有无批量限制、有无流式处理  
**审查方法**：检查批量处理循环（逐张 vs 全量加载）、Excel 读取模式（read_only vs 全量）、ClosedXML 内存模型

---

## 发现

### 🟢 plot_curves 批量出图：串行渲染 + 逐张 GC

```python
for i, job in enumerate(jobs, start=1):
    written.append(render_plot_to_png(job))  # 一张一张来
```

- 每次只有 1 个 matplotlib figure 在内存中
- `render_plot_to_png` 内部 `fig.clf()` + `plt.close()` 释放
- 单张失败不打断批量 ✅
- 进度回调异常被吞，不拖垮批量 ✅

**100 张图也没问题**——时间是瓶颈但内存不是。

### 🟡 Excel 读取：全量加载到内存（excel_reader.py:160）

```python
all_rows = list(ws.iter_rows(values_only=True))  # 全部行一次性加载
```

openpyxl 的 `read_only` 模式支持流式迭代，但这里 `list()` 强制全量加载。对于土木检测 Excel（通常 50-500 行），完全没问题。但如果有人误拖了一个 100 万行的 CSV 转的 xlsx，内存会暴涨。

**无文件大小/行数上限检查。**

### 🟡 C# ClosedXML：全量 workbook 加载

`XlsxHandlers.cs:74` 和 `AnchorHandlers.cs:127` 用 `new XLWorkbook(path)` 加载整个工作簿。ClosedXML 不支持 read-only 模式——整个 workbook 的生命周期都在内存中。对于追加 sheet 场景（已有文件→读→追→写），旧文件 + 新数据同时在内存。

典型工程检测 xlsx（10-50 sheets × 100 rows = ~500KB）完全可承受，但无上限守卫。

### 🟢 config.toml 有 max_concurrent 限制

```toml
max_concurrent = 1  # Word COM 串行运行避免争抢
```

`configs/loader.py:116` 校验 `1 <= max_concurrent <= 8`。Word COM 并发数已限制，防止多个 Word 实例同时跑撑爆内存。

### 🟢 SHA256 计算：1MB 分块流式

`compute_file_sha256` 用 `chunk_size=1<<20` 分块读取——即使是几百 MB 的大文件也不会一次性吃光内存。

### 🟢 无网络场景——无内存泄漏来源

纯本地应用，无 WebSocket/HTTP 长连接，无 Service Worker 缓存累积。没有常见的内存泄漏来源。

## 评级

| 问题 | 严重度 | 说明 |
|------|--------|------|
| plot_curves 串行渲染 | 🟢 | 1 张图/次，不会 OOM |
| Excel 全量加载 | 🟡 | 正常数据量安全，缺上限守卫 |
| ClosedXML 全量加载 | 🟡 | 同上 |
| max_concurrent 限制 | 🟢 | Word COM 已限制 |
| SHA256 流式 | 🟢 | 大文件友好 |

---

## 建议

1. **加文件大小/行数预警**：Excel 读取前检查文件大小，>50MB 或行数 >10 万行时前端警告「文件较大，处理可能需要较长时间」
2. **plot_curves 加 max_jobs 限制**：确保不会生成数千张图
3. **C# sidecar 加 GC 提示**：大批量处理后手动触发 `GC.Collect()`（ClosedXML 容易残留 COM 对象）
