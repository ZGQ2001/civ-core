## 一、项目定位与系统边界

土木工程领域（道路桥梁、房屋检测与鉴定）的检测报告编制繁琐易错。本工具定位为**内业报告自动化辅助**，不替代仪器自带专业分析软件，不负责复杂 CAD 图纸生产。

**核心业务**：接收仪器导出的现成 Excel/CSV 数据，聚焦 Word 和 Excel 的高效流转、规范套用、报告生产以及报告编写过程中其他需要的工具。

**标准内业流程**（每个程序均可单独操作）：

```
仪器导出数据 → 导入工具 → 数据格式化与规范评定 →
（按需生成辅助图表）→ 基于模板自动填充报告 → 人工校核 → 自动化排版
```

### 1.1 渐进式重构原则

本工具不是一次性从零设计的大系统，而是**对现有可工作脚本的逐步规范化整合**。开发顺序遵循以下铁律：

1. **不在没有真实业务的情况下凭空设计抽象**——避免 premature abstraction
2. **每次只重构一个脚本到新架构下**，跑通后再接下一个
3. **22 条原则不是一次性满足清单**，是**逐条往上贴的检查表**
4. **MVP 阶段不碰的能力**（UI、预览、撤销、Word 输出）一律推迟

## 二、技术栈决策

下表为已锁定决策，未经评审不得变更：

| 维度 | 选型 | 关键依据 |
|------|------|---------|
| Python 版本 | 3.12+ | 利用 `tomllib` 标准库读 toml |
| UI 框架 | PySide6 + qfluentwidgets | ⚠️ qfluentwidgets 双重许可（GPLv3 / 商业），**内部自用**前提下使用，未来若分发必须重评估 |
| Excel 操作 | openpyxl | 纯 Python、xlsx 原生；不引入 pandas |
| Word 操作（主） | docxtpl + python-docx | docxtpl 处理模板化变量替换/循环/表格/图片，python-docx 处理需直接操作 XML 的精细场景 |
| Word 操作（辅，受限） | pywin32 COM | **仅用于域刷新、目录更新、最终格式精修等 docxtpl 无法完成的后处理步骤**。详见 §2.2 |
| 图表 | matplotlib | 必配：中文字体（YaHei/SimHei）、`io.BytesIO` 内存渲染、DPI ≥ 300、关闭 unicode 负号显示 |
| 配置文件 | **toml**（读：`tomllib`；写：`tomli-w`） | **不是 yaml**。toml 与 `pyproject.toml` 同生态，类型严格，不易出隐性 bug |
| 数据契约 | `@dataclass` + 自定义 validator | **不引入 pydantic**。详见 §2.3 |
| 撤销引擎 | `QUndoStack` + 双层命令模式 | Core 层纯 Python 命令对象（可单元测试、无 Qt 依赖），UI 层 `QUndoCommand` 薄包装 |
| 日志 | 标准库 `logging` | 文件 + 控制台双 handler；时间统一带时区（UTC → Asia/Shanghai） |
| 类型检查 | Pylance Basic | **零红线提交**为硬性门槛 |
| 单元测试 | pytest | 测试目录与源码目录平级 |

### 2.1 依赖最小化白名单

**运行依赖白名单**（新增需评审）：

- PySide6、PySide6-Fluent-Widgets
- openpyxl、docxtpl、python-docx
- matplotlib
- tomli-w（写 toml）
- **pywin32**（仅限 §2.2 列出的用途）
- **pymupdf**（PDF 处理，限定 `core/pdf_*` 模块使用）

**严禁引入**：

- pandas、numpy（除非 matplotlib 强制传递依赖）
- pydantic（理由见 §2.3）
- requests（除非真有 HTTP 需求）

### 2.2 pywin32 / COM 使用约束（v2.2 新增）

本节为 v2.2 重要决策，因现有代码 `io/word_app.py` 已大量使用 COM。

**允许使用 COM 的场景**：

| 场景 | 理由 |
|------|------|
| Word 域代码刷新（`Fields.Update`） | docxtpl/python-docx 不支持域 |
| 目录/索引重建（`TablesOfContents.Update`） | 同上 |
| 文档转 PDF（`SaveAs2`） | python-docx 无 PDF 输出能力 |
| 最终格式精修（页眉页脚同步、断页修正） | 无可替代方案 |

**禁止使用 COM 的场景**：

- 模板变量替换（必须用 docxtpl）
- 表格批量插入（必须用 docxtpl 的 `tpl.subdoc` 或 python-docx）
- 图片插入（必须用 docxtpl 的 `InlineImage`）
- 任何在无 Word 安装环境下也应能工作的逻辑

**封装要求**：

- 所有 COM 调用集中在 `infra_io/word_com.py` 单一模块
- 该模块必须提供"COM 不可用"的降级路径（如：仅生成 docx，跳过域刷新，给出警告）
- COM 操作必须包装在 `try/finally` 中确保进程释放
- COM 调用前必须通过 §3.0 P0-4 文件占用预检

**WPS 兼容性**：COM 路径不兼容 WPS。涉及 COM 的功能必须在 UI 上显式标注"仅 Microsoft Word 可用"，并提供 docxtpl-only 的备选生产线。

### 2.3 不引入 pydantic 的理由（v2.2 新增）

现有代码已使用 pydantic，本次决策为**移除**：

1. **依赖体积**：pydantic v2 携带 `pydantic-core`（Rust 编译产物），打包体积显著
2. **学习成本**：团队后续维护者需多掌握一套校验 DSL
3. **MVP 校验需求简单**：清洗管线 + dataclass `__post_init__` + `utils/validators.py` 足够覆盖
4. **类型契约清晰**：dataclass 是 Python 标准，IDE/Pylance 支持开箱即用

**统一校验模式**（实现规范）：

```python
@dataclass(frozen=True)
class ChartTask:
    title: str
    x_label: str
    y_label: str
    series: list[ScatterSeries]
    output_path: Path

    def __post_init__(self) -> None:
        if not self.title.strip():
            raise ValueError("title 不可为空白")
        if not self.series:
            raise ValueError("series 至少含一条")
        # 跨字段约束放这里
```

复杂跨字段校验放 `utils/validators.py`，由 IO 层在装载前调用。

## 三、架构原则分档

原 22 条原则按落地阶段分三档。**MVP 阶段只需满足 P0 档**，P1/P2 在对应阶段强制启用。

### 3.0 P0 档：MVP 必须落地（10 条）

> 不论做什么工具都必须遵守的工程底线。第一个 MVP 工具就要全部命中。

#### P0-1 全局配置外部化（原 3.13）

规范系数、阈值、图表色板、Word 书签名、模板路径等可变参数，全部抽离至外部 toml 配置。**程序源码中禁止出现硬编码业务常量字符串或数值**。配置加载时进行类型校验和范围检查，非法值在启动阶段拒绝并报错。

#### P0-2 严格 MVC 分层（原 3.16，强化）

`ui/` 目录代码仅负责控件布局、事件绑定和信号转发。**绝对禁止**在 `on_button_clicked` 等槽函数中直接调用 `openpyxl`、`python-docx`、`pathlib` 等 IO 或文件操作。所有业务动作必须通过 `core/` 层暴露的纯函数接口完成，由 IO 层在后台线程中执行。

> MVP 阶段虽无 UI，本条仍生效——业务函数禁止直接读写文件，文件 IO 由 `infra_io/` 层独立模块负责。

#### P0-3 IO 输入清洗隔离带（原 3.20，强化）

`infra_io/` 层是所有外部数据的唯一入口。原始数据进入 `core/` 层前，**必须**经过强制性清洗管线：去除首尾空白、全角字符转半角、空行剔除、`NaN` / `NaT` 替换为显式哨兵值。清洗后数据**必须**成功装载至 `domain/schema.py` 定义的 `@dataclass` 实例中，未通过验证的数据行记录日志后丢弃。

**Core 层函数在类型注解层面拒绝接受 `dict | None` 等模糊类型**——参数类型必须是具体的 dataclass。

#### P0-4 文件占用预检（原 3.18，精确化）

执行覆盖保存或导出操作前，**必须通过操作系统级文件锁**检测目标文件是否被外部进程占用。检测到占用则阻塞写入并提示：

```
检测到 <完整路径> 正在被 <进程名> 使用，请关闭该文件后重试。
```

**实现规范（v2.2 明确）**：

- 主策略：尝试 `open(path, 'r+b')` 捕获 `PermissionError`（跨平台、零依赖）
- 进程名探测：可选，若引入 psutil 则需评审。**MVP 阶段允许不显示进程名**，提示语降级为"文件正在被其他程序使用"
- 检测窗口：仅在写入瞬时检测，**不做轮询监控**

#### P0-5 原子写入策略（原 3.19，v2.2 修订）

所有输出操作遵循三阶段提交：

1. **在目标文件同目录下**生成临时文件（命名形如 `.<filename>.tmp.<uuid>`）
2. 校验输出完整性（大小 > 0、可成功打开）
3. 确认目标路径不存在文件锁后，通过 `os.replace` 原子移动落盘
4. **跨设备/盘符 fallback**：若 `os.replace` 抛 `OSError`，改用 `shutil.move` 退化为非原子复制+删除

> **v2.2 修订说明**：v2.1 写的"系统临时目录"在 Windows 多盘符场景下会触发 `OSError: [WinError 17]`。改为目标文件同目录后，`os.replace` 在同一文件系统内可保证原子性。

**实现示例**：

```python
def atomic_write_bytes(target: Path, payload: bytes) -> None:
    target.parent.mkdir(parents=True, exist_ok=True)
    tmp = target.parent / f".{target.name}.tmp.{uuid.uuid4().hex}"
    try:
        tmp.write_bytes(payload)
        if tmp.stat().st_size == 0:
            raise IOError("生成的临时文件为空")
        try:
            os.replace(tmp, target)
        except OSError:
            shutil.move(str(tmp), str(target))
    finally:
        if tmp.exists():
            tmp.unlink(missing_ok=True)
```

任何阶段失败均不产生残缺的目标文件。

#### P0-6 输出幂等性保证（原 3.22，v2.2 修订）

相同输入（原始数据 + 配置快照）多次执行**应产生业务等价输出**：

- 每次生成前清理目标区域或重建输出文件，**禁止追加写入**
- 涉及随机性的操作（如图表自动配色）使用确定性 seed
- 生成时间戳仅写入文件属性，**不进入业务输出正文**

**例外条款（v2.2 新增）**：

| 输出类型 | 一致性级别 | 说明 |
|---------|-----------|------|
| PNG / JPEG（matplotlib） | **字节级一致** | 固定 seed，`rcParams['svg.hashsalt']` 等 |
| CSV / TXT | **字节级一致** | 显式控制行尾、编码、BOM |
| docx / xlsx（OOXML） | **业务内容一致** | OOXML 是 ZIP+XML，允许 XML 属性顺序、ZIP entry 顺序、`docProps` 元数据时间戳的微小差异。验证手段为：解压后比较 `word/document.xml` 的语义内容 |
| pdf | **业务内容一致** | 同上，PDF 内嵌 `/CreationDate` |

> **v2.2 修订说明**：v2.1 要求所有输出"字节级一致"对 OOXML/PDF 不可达成。改为分层要求后，PNG/CSV 严格执行，Word/Excel 落到业务等价层面。

#### P0-7 异常提示标准格式（原 3.10）

异常呈现三段式：**故障定位** → **原因诊断** → **修复建议**。

- **MVP（CLI）阶段**：通过日志和标准错误流以三段式格式输出
- **V1（GUI）阶段**：通过 `qfluentwidgets.InfoBar` 组件呈现

**禁止把原始 traceback 或异常类名直接抛给用户**——traceback 应进日志文件供事后分析。

#### P0-8 数据溯源与审计日志（原 3.12）

每次运行生成一条审计日志记录：时间戳（ISO 8601 + 时区）、输入文件标识（路径 + SHA256）、执行参数快照、输出文件路径。

**原始导入数据（`data/raw/`）与处理后数据（`data/output/`）物理隔离，处理过程不可逆写原始文件。**

#### P0-9 批量处理优先原则（原 3.3，精确化）

底层函数签名**强制接受 `List[T]` 形式的批量输入**。**禁止在业务层或 UI 层以 `for` 循环逐条调用单例处理函数**。批量语义在 IO 层出口处一次性完成。

#### P0-10 参数自定义扩展（原 3.15）

每个关键参数同时支持：

- 系统内置预设值（标注来源规范编号）
- 用户自定义输入

切换预设时保留用户上次自定义值。

### 3.1 P1 档：UI 接入阶段强制（8 条）

> 第一个工具有 UI 时全部启用。MVP（CLI）暂不要求。

#### P1-1 领域术语强制规范（原 3.1）

UI 组件标识、底层参数命名、日志输出及异常信息中，禁止出现通用编程术语。**桩号、里程、脱空、异常值剔除**等土木工程专有名词为唯一合法表述。布尔值、数组、键值对等计算机科学词汇视为不合规，提交前必须完成领域化重命名。

#### P1-2 强制输出前预览（原 3.5）

排版、图表嵌入、模板填充操作，落盘前**必须经过轻量级预览流渲染**。预览流与最终输出**使用同一套渲染管线**，确保所见即所得。

#### P1-3 渐进式引导机制（原 3.6）

每个功能模块面板提供"生成示例模板"按钮，一键加载符合当前模块输入契约的示例数据。复杂参数表单项旁强制配备 `[?]` 悬停提示组件，提示内容须包含：参数含义、默认值来源、修改后果。

#### P1-4 工作流结构布局（原 3.7）

主界面导航结构按内业标准流程组织：`数据接入 → 格式化/评定 → 报告生成 → 校核 → 终版输出`。**菜单层级 ≤ 3 级**，同级菜单项按执行顺序排列，**不得按字母或功能类别排序**。

#### P1-5 异步执行与 UI 线程保护（原 3.8，精确化）

任何涉及磁盘 IO 或 CPU 密集计算的代码路径，**严禁在主线程（事件循环线程）中同步执行**。必须通过 **`QThreadPool + QRunnable`** 投递至工作线程，并通过 Signal/Slot 机制向 UI 层回传进度与结果。

**UI 线程任何超 50ms 的阻塞即视为缺陷。**

> **COM 调用特别约束（v2.2）**：pywin32 COM 调用属于跨进程同步操作，必须通过 `QThreadPool` 投递至工作线程。COM 线程必须调用 `pythoncom.CoInitialize()` / `CoUninitialize()` 配对。

#### P1-6 全局撤销栈与自动备份（原 3.9，精确化）

核心数据变更操作（表格编辑、参数修改、批量替换）必须注册到 `QUndoStack`，支持 **Ctrl+Z / Ctrl+Y** 业务级撤销与重做。

涉及覆写已有文件的操作，执行前自动在临时目录生成 **`.bak` 快照文件**，保留**最近 3 个版本**。

> 实现遵循 §2 决策：Core 层纯 Python 命令对象 + UI 层 `QUndoCommand` 薄包装。

#### P1-7 评定算法白盒化（原 3.11）

评定计算模块的数值修约规则、有效位数、舍入模式必须在代码中以**显式函数封装**，**禁止依赖浮点默认行为**。所有中间计算步骤可通过日志级别 `DEBUG` 逐行输出，便于人工验算。

#### P1-8 工程数据可视化规范（原 3.17，精确化）

表格视图强制使用等宽字体（**Consolas / Source Code Pro**）。数据列中：

- 超出规范阈值的单元格 → **红色前景色**高亮
- 低于下限 → **蓝色前景色**高亮
- 非异常数据 → **中性灰色**，减少视觉干扰

### 3.2 P2 档：多模块/插件化阶段（4 条）

> 工具数量超过 3 个、需要插件化承载多检测类型时启用。

#### P2-1 规范预置与约定优于配置（原 3.2）

系统出厂时内置现行国标（GB）及地方标准（DB）的默认参数集。常见检测类型提供"一键符合 XX 规范"的预设配置入口。用户无需逐项调参即可获得符合规范输出，仅在需要偏离时有选择性覆盖。

#### P2-2 数据链路端到端联动（原 3.4，精确化）

建立 `Excel 数据源 → 评定计算 → 图表渲染 → Word 报告` 的单向可追溯链路。源数据变更后，下游图表和报告字段**支持按脏标记增量重绘**，不得要求用户手动重新执行完整流程。提供"一键清理空行"、"一键更新域/目录"等复合批处理指令。

#### P2-3 检测类型插件化架构（原 3.14，精确化）

核心框架仅提供底座能力（配置加载、日志、调度、UI 框架）。道路检测、桥梁检测、房屋鉴定等具体业务作为独立 Python 子包挂载，**需实现约定的 Plugin 接口**。加载/卸载不要求重启应用。

> Plugin 接口的具体方法签名在进入 P2 阶段前由专题评审定义。

#### P2-4 局部降级与醒目占位（原 3.21）

模板填充过程中遇到缺失书签、数据字段缺值等情况，**禁止抛出异常中断整体生成**。必须在日志中以 `WARNING` 级别记录缺失项，并在 Word 文档对应位置插入红色高亮文本：

```
[系统警告：此处缺少 <字段名>]
```

确保校核人员在报告中一眼定位问题，手动补填。

## 四、强制性工程与编码规范

任何代码必须满足以下底线：

### 4.1 架构与契约

- **架构解耦**：Core / IO / UI 完全分离
- **契约与类型**：全开类型注解（Type Hints）。模块间数据流转**必须使用 `@dataclass`，严禁裸传字典**
- **静态检查**：所有模块通过 Pylance **Basic** 级别检查，**零红线提交**

### 4.2 安全与防错

- 资源必须用 `with` 管理
- 严禁 `except: pass` 吞咽异常；报错必须带上下文变量
- 系统命令安全：严禁拼接 Shell 字符串，必须用 `subprocess.run([...])` 列表形式
- **COM 资源管理（v2.2）**：`win32com` 客户端必须 `try/finally` 包裹，调用 `Quit()` 显式释放

### 4.3 跨平台与编码

- 所有路径必须用 `pathlib.Path`，**禁止字符串拼接**
- 所有文件读写**必须显式指定** `encoding='utf-8'`
- COM 路径仅限 Windows + Office；非 COM 路径必须保持跨平台

### 4.4 时间与日志

- 时间处理必须带时区（推荐 `datetime.now(ZoneInfo("Asia/Shanghai"))`）
- 严禁仅用 `print`，关键节点必须 `logging` 记录到文件
- 日志格式建议：`时间 | 级别 | 模块 | 函数 | 消息 + 上下文变量`

### 4.5 依赖与交付

- 优先标准库；引入第三方依赖必须在 §2.1 白名单内或经评审
- **严禁一次性输出所有代码**——分模块分阶段交付，每个模块独立可测

## 八、V1 阶段：PySide6 集成规范

> **MVP 阶段不启用本章**。第一个 GUI 工具开工前对照此节实施。

### 8.1 控件选型

| 功能 | 实现方案 |
|------|---------|
| 工具分类树 + 搜索框 | `qfluentwidgets.NavigationView` 或 `TreeWidget` |
| 工具详情页切换 | `QStackedWidget` 或路由机制 |
| 参数配置面板 | `QFormLayout` 动态读取 `domain/schema.py` 自动生成输入框 |
| 日志面板 | `QPlainTextEdit` + 自定义 `logging.Handler`，INFO/WARNING/ERROR 彩色高亮，支持 Ctrl+F 搜索和右键复制 |
| 表格编辑器 | `QTableView` + 自定义 `QAbstractTableModel`（数据/视图分离） |
| 图表预览 | `matplotlib.backends.backend_qtagg` 或 `PySide6.QtCharts` |
| 撤销引擎 | `QUndoStack` + `QUndoCommand`，绑定 Ctrl+Z / Ctrl+Y |
| 批量任务 | `QListWidget` 或表格展示进度，全局 Flag 投递暂停/继续/跳过信号到工作线程 |
| 错误提示 | `qfluentwidgets.InfoBar`（中文友好提示 + 修复建议） |
| 进度状态 | `QStatusBar` + 环形/条形进度组件，绑定线程池 Signal |

### 8.2 线程模型

- **严禁在 UI 线程执行 IO 或密集计算**
- 必须使用 `QThreadPool + QRunnable` 模式
- Signal/Slot 与主线程通信
- **COM 调用线程必须 `pythoncom.CoInitialize()` / `CoUninitialize()` 配对（v2.2）**
- UI 线程任何超 **50ms** 的阻塞即视为缺陷（参 P1-5）

### 8.3 与 P1 档原则的对应关系

本节实现是 P1-1 至 P1-8 在 PySide6 框架下的具体落地形式。任何 V1 阶段的代码必须同时满足两边。
