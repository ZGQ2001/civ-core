# Excel 驱动模板（备选方案，待评估）

> **状态**：构思，未实施。当前 PR #8 已交付 Word 占位符方案，本想法是更进一步的简化。
> **日期**：2026-05-26
> **背景**：在 [2026-05-26 跟用户对话](#) 中提出，作为现有 Word 占位符方案的可能替代。

---

## 一、核心想法

用户的真实操作习惯是 Excel，不是 Word。提案：

```
输入 Excel（原始数据）
       ↓ anchor.run 计算
输出 Excel（含 3 sheet）：
  ├─ 计算数据表 sheet：每根锚杆一行，表头 = {{占位符}} 形式
  ├─ 项目信息表 sheet：单行 / 单列，字段名 = {{占位符}}
  └─ 说明 sheet：可用占位符清单 + 使用提示
```

**关键设计点**：

1. **表头即占位符**：用户在第一行写 `{{anchor_id}}` `{{弹性位移量}}`，程序按表头匹配字段填值
2. **用户可自由增减字段**：加列 = 加字段；删列 = 不要那字段；不需要管"程序里有没有定义"
3. **双花括号** `{{...}}`：比 `{...}` 更不容易跟普通文字混淆
4. **中英文标点容错**：识别 `（）` `()` `，` `,` `：` `:` 等等价对，防用户敲错
5. **合并单元格保留**：用户在 Excel 里做的合并格状态保留，程序只替换文字内容

## 二、对比已有方案

| 维度 | Word 占位符（当前 PR #8） | Excel 模板（本提案） |
|---|---|---|

## 三、待解决的难点

### 3.1 「数据表头 = 模板占位符」一处定义

**核心约束**（用户 2026-05-26 补充）：
- 程序生成的「计算数据表」表头本身就**可以直接复制**去做模板（同一套字段名 = 同一套占位符）
- 用户加一列/删一列时，**程序不该报错也不该挂**（缺失字段 → 留原文或 `/`，多余字段 → 引擎不认识就忽略）
- 用户的体感：「程序一直能用，加减字段不影响」

实施上：
- `AnchorFieldCatalog.Name` 字段就是表头中文名；`Key` 是引擎主键
- 模板支持 `{{中文名}}` 和 `{{key}}` 都能识别（占位符引擎已支持）
- 程序生成数据 Excel 时表头用 `Name`，用户复制 `Name` 去模板 = 直接能用

### 3.2 合并单元格 —— 两种来源

| 来源 | 例子 | 难度 |
|---|---|---|
| **用户合并** | 排版需求：表头跨列、左侧"批次"跨行 | 静态、易处理（保留主格、读写忽略覆盖格） |
| **程序合并** | 算前 N 行平均值时跨 N 行合并；同表里既有"前 5 行平均"又有"前 3 行平均" | **动态**：行数由数据驱动，模板里不好预定义 |

**程序合并的特殊性**：
- 不能用纯占位符表达（占位符是「这一格填啥值」，不是「这一区聚合后填啥」）
- 思路 A：模板里写聚合占位符如 `{{avg(disp, 5)}}`（引入函数式占位符 —— 引擎复杂度上升）
- 思路 B：程序按固定布局产出 Excel（含程序自己合并好的格），用户复制这份 Excel 当模板 —— 跟 3.1 的「数据表头即模板」呼应

倾向思路 B —— 因为它跟 3.1 的设计哲学一致：程序产出的就是用户能直接用的。

### 3.3 空单元格用 `/` 占位

土木报告习惯：「无」「不适用」「待补」全部用 `/` 表达，不留空。

实施细节：
- Resolver 返回 null（数据缺失）→ 引擎写 `/`
- 占位符不在 catalog 里（用户敲错）→ 留原文 `{{xxx}}`（防止吞没）
- 区分这两种情况：缺数据 vs 拼错字段，分别处理

C# 字符串处理对 `/` 这种内容免疫，实施上没什么风险。

### 3.4 中英文标点容错

用户敲 `{{弹性位移量 (mm)}}` 时，括号/逗号/冒号可能用中文符。需要在匹配前归一化：

```
（）→ ()    ，→ ,    ：→ :    " " → 半角空格
```

正则归一化 raw 占位符 + 同样归一化 catalog 的 Name；归一后比较。

### 3.5 不规则合并单元格

「不规则合并」（A1:B2 + B3:C3 + ...）让表头不对齐数据列。

**思路**：
- 合并区域只取左上角主格当占位符匹配点
- 数据写入也只写主格
- 跟 3.2 思路 B 配合：程序产出的 Excel 自己保证规则合并，用户少改

### 3.2 中英文标点容错

用户敲 `{{弹性位移量 (mm)}}` 时，括号可能是 `(` / `（` / 其他。需要在占位符匹配前归一化：

```
({|（) → {
(}|）) → }
(\(|（) → (    # 内部内容里的括号统一
(\)|）) → )
(,|，) → ,
(:|：) → :
```

实现：正则 + 归一化函数处理 raw 文本，匹配字典的 key/name 也做同样归一化。

### 3.3 "项目信息表"该是哪种排版？

两种常见排版：
- A. 横排：第一行表头 `{{client_name}} {{project_name}} ...`，第二行数据
- B. 竖排：第一列字段名 `委托方/工程名/...`，第二列对应值

实际项目信息表通常是竖排（B 排版），数据表是横排（A 排版）。需要程序自动判断或用户在 sheet 名里标注。

## 五、不做的判断

**暂不实施的理由**：
- 当前 PR #8 已经把 Word 占位符路径做完了，能跑
- Excel 模板真正的工程量集中在合并单元格 + 标点容错，没明显比 Word 路径短
- 用户的 mental model 还在演化（Word 编辑器 → Word 占位符 → Excel 模板这条路径走了 3 步）；再让方案演化一段时间再下手

**触发条件**（满足任一即可启动实施）：
- 用户拿到 Word 占位符路径跑过一两次真实甲方报告后，明确说"我宁可用 Excel"
- 出现真实使用障碍（比如 Word 模板里跨 Run 拆分严重导致体验差）
- 实施成本估算降到 ≤ 1 天（解决合并单元格 + 标点容错后）
