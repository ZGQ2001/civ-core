> **日期**：2026-05-26

## 一、核心想法

用户的真实操作习惯是 Excel，不是 Word。提案：

```
输入 Excel（原始数据）
       ↓ anchor.run 计算
输出 Excel（含 3 sheet）：
  ├─ 计算数据表 sheet：每根锚杆一行，表头 = {{占位符}} 形式
  ├─ 项目信息表 sheet：单行 / 单列，字段名 = {{占位符}}
  └─ 说明 sheet：可用占位符清单 + 使用提示
```

**关键设计点**：

1. **表头即占位符**：用户在第一行写 `{{anchor_id}}` `{{弹性位移量}}`，程序按表头匹配字段填值
2. **用户可自由增减字段**：加列 = 加字段；删列 = 不要那字段；不需要管"程序里有没有定义"
3. **双花括号** `{{...}}`：比 `{...}` 更不容易跟普通文字混淆
4. **中英文标点容错**：识别 `（）` `()` `，` `,` `：` `:` 等等价对，防用户敲错
5. **合并单元格保留**：用户在 Excel 里做的合并格状态保留，程序只替换文字内容

## 三、待解决的难点

### 3.1 不规则合并单元格

如果用户的 Excel 模板含「不规则合并」（A1:B2 合并 + B3:C3 合并 + ...），程序填值时按表头匹配会困难——
表头不对齐数据列。

**初步思路**：
- 不规则合并区域里只取「左上角主格」的文本当占位符
- 主格 = 占位符匹配点；其他被覆盖的格自动跳过
- 数据写入也只写主格

### 3.2 中英文标点容错

用户敲 `{{弹性位移量 (mm)}}` 时，括号可能是 `(` / `（` / 其他。需要在占位符匹配前归一化：

```
({|（) → {
(}|）) → }
(\(|（) → (    # 内部内容里的括号统一
(\)|）) → )
(,|，) → ,
(:|：) → :
```

实现：正则 + 归一化函数处理 raw 文本，匹配字典的 key/name 也做同样归一化。

### 3.3 "项目信息表"该是哪种排版？

两种常见排版：
- A. 横排：第一行表头 `{{client_name}} {{project_name}} ...`，第二行数据
- B. 竖排：第一列字段名 `委托方/工程名/...`，第二列对应值

实际项目信息表通常是竖排（B 排版），数据表是横排（A 排版）。需要程序自动判断或用户在 sheet 名里标注。

## 四、跟当前 PR #8 的关系

**保留 PR #8 的 Word 路径**：
- 已交付完整测试，能用
- 部分甲方明确要求 docx 报告

**Excel 模板路径以后做（如果真有需求）**：
- 用户实际用 Excel 用得更顺时再实施
- 实施时不需要保留 Word 路径（直接替换或并存）

**模板编辑器前端工具页可能整个删掉**：
- 字段对照表的能力可以挪到「数据处理」工具页的旁边作为辅助信息块
- 减少一个 ActivityBar 入口

## 五、不做的判断

**暂不实施的理由**：
- 当前 PR #8 已经把 Word 占位符路径做完了，能跑
- Excel 模板真正的工程量集中在合并单元格 + 标点容错，没明显比 Word 路径短
- 用户的 mental model 还在演化（Word 编辑器 → Word 占位符 → Excel 模板这条路径走了 3 步）；再让方案演化一段时间再下手

**触发条件**（满足任一即可启动实施）：
- 用户拿到 Word 占位符路径跑过一两次真实甲方报告后，明确说"我宁可用 Excel"
- 出现真实使用障碍（比如 Word 模板里跨 Run 拆分严重导致体验差）
- 实施成本估算降到 ≤ 1 天（解决合并单元格 + 标点容错后）
