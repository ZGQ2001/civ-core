## 一、你定的四条原则（拿来当验收标准）

1. **用最少的东西，实现最多的功能**
2. **好维护**
3. **必须可靠**
4. **同一类型的功能、不同内容，要有统一意识**

下面所有事，都对着这四条来。

---

## 二、现在的毛病（一句话）

锚杆、防火涂层、里氏硬度，本该是**同一条流水线上、换不同"料"**的三种活儿。可现在是**三条并排的半独立产线**：锚杆那条最齐全，防火涂层是照着锚杆**另抄了一条短的**，里氏抄了半条。

这一条就把你四条全踩了：

- 不够**省**——同一段小活儿抄了好几份（比如给表格起名字那段，抄了四份）。
- 不**好维护**——改一个地方要追着改三四份，容易漏。
- 不**可靠**——抄出来的副本各自跑偏，一份修了别份没修；最怕同一种判定在不同类型里算得不一样。
- 不**统一**——三种检测，操作起来三种手感。

---

## 三、治法（一句话）

**三条并一条。** 把"读表 → 套规范 → 出表 → 出报告"这套**搬运的架子**只造一次、大家共用；每个检测类型只交自己的**料**——它的判定公式、它要填的字段。

以后加钻芯、回弹，是**填料，不是重造线**。

这样正好对上四条：架子只一条（省）、改一处（好维护）、只有一条久经测试的线（可靠）、人人走同一条（统一）。

---

### 第 0 步细节（已经在做的）

- ✅ 删掉没人用的死代码（Python，本地测试 226 项全过）。
- ✅ 合并"给表格起名字"那段重复代码（四份 → 一份，行为不变）。
- ⬜ 还有几处近似的小重复（解析用户输入、列名规整等）——**必须逐一核对是不是真的一模一样再合**，不一样的就先留着。（已发现"解析用户输入"那两份其实**不一样**，不能盲并——这正说明为什么要一条条核。）

### 第 1 步细节（防火出报告不重算）

数据处理那步已经把数算好、存进了结果表。出报告本该直接读，不该再算一遍：

- 锚杆：出报告走"读结果"的路（`AnchorResultReader`），**不重算**。
- 防火：**没有这条路**，出报告只能拿原始表重新算一遍——白算、还多一条要维护的路、万一改过结果它会无视。
- 做法：给防火补一个"读结果"的路（照锚杆的样），让它出报告也直接读数据处理存下的结果。对上"省 + 可靠 + 统一"。

### 第 2 步细节（录入向"全在表里"收敛 + 界面拉齐）

防火"信息都在数据表里"是好样板，锚杆向它看齐：

- 锚杆那套额外的"批次参数"UI 表单是多余的（参数本来就能写进表里的"批次信息"sheet）→ 逐步去掉，统一从表里读。
- 报告类型从**写死的三选一** → **勾选框**（后端早支持任意组合）。
- "检测项目"下拉现在**只有锚杆**（系统初始化只建了锚杆目录）→ 补齐所有类型。

## 六、已拍板（2026-06-02）

1. **出报告不重算。** 数据处理已经算过，出报告就读已算好的结果。现状：锚杆读结果（有 `AnchorResultReader`），**防火却重算**（没有"读结果"的路）。定了：所有类型出报告都读结果——省、可靠（只有一份结果，不会两条路算出不一样）、统一。
2. **录入向防火看齐。** 不给防火补表单；"信息全在数据表里"是好样板，**锚杆向它收敛**（去掉多余的 UI 参数表单）。
3. **短期不加新检测类型。** 所以不提前造"为加类型省事"的大抽象（违背"用最少"）；只干对现有两种就有用的事。
